# MFA API Implementation Verification Specification

This document provides a comprehensive specification for verifying the Multi-Factor Authentication (MFA) endpoints implementation in the Teal Payroll API backend code.

## Overview

Three new MFA endpoints have been added to the members system:
1. `POST /members/mfa/change` - Generate TOTP secret
2. `POST /members/mfa/change-verify` - Verify and enable MFA
3. `POST /members/mfa/validate` - Validate MFA during login

Additionally, the existing `POST /members/signin` endpoint has been updated to support MFA flow.

---

## Endpoint 1: POST /members/mfa/change

### Purpose
Generate a TOTP (Time-Based One-Time Password) secret for a member to enable MFA.

### Authentication Required
- **Type:** Bearer token (JWT)
- **Header:** `Authorization: Bearer <jwt_token>`
- **Security Scheme:** MemberBearerAuth

### Request Specification

#### Headers
```
Authorization: Bearer <jwt_token>
Content-Type: application/json
```

#### Request Body (JSON)
```json
{
  "access_token": "string (required)"
}
```

**Field Validation:**
- `access_token`: 
  - Type: string
  - Required: YES
  - Source: Must be the JWT token from `POST /members/signin` response (the `jwt` field)
  - Validation: Should be the same value as the Bearer token in the Authorization header

### Response Specification

#### Success Response (200 OK)
```json
{
  "secret": "string"
}
```

**Field Details:**
- `secret`:
  - Type: string
  - Format: Base32-encoded TOTP secret seed
  - Example: "JBSWY3DPEHPK3PXP"
  - Usage: Used to generate QR code or for manual entry into authenticator apps

#### Error Responses

**400 Bad Request**
- Trigger: Missing `access_token` field in request body
- Response: Standard error format with error messages array

**401 Unauthorized**
- Trigger: Invalid or expired JWT token (either in Authorization header or access_token field)
- Response: Standard unauthorized error format

### Backend Verification Checklist
- [ ] Endpoint exists at route `POST /members/mfa/change`
- [ ] Requires authentication via Bearer token
- [ ] Validates `access_token` field is present in request body
- [ ] Validates JWT token is valid and not expired
- [ ] Generates a valid Base32-encoded TOTP secret
- [ ] Returns 200 with `secret` field on success
- [ ] Returns 400 if `access_token` is missing
- [ ] Returns 401 if token is invalid/expired

## Endpoint 2: POST /members/mfa/change-verify

### Purpose
Verify the TOTP code from the member's authenticator app and permanently enable MFA for their account.

### Authentication Required
- **Type:** Bearer token (JWT)
- **Header:** `Authorization: Bearer <jwt_token>`
- **Security Scheme:** MemberBearerAuth

### Request Specification

#### Headers
```
Authorization: Bearer <jwt_token>
X-Member-Username: <email> (required)
Content-Type: application/json
```

**Header Validation:**
- `X-Member-Username`:
  - Type: string
  - Format: email
  - Required: YES
  - Example: "john.smith@goteal.co"

#### Request Body (JSON)
```json
{
  "user_code": "string (required)",
  "access_token": "string (required)"
}
```

**Field Validation:**
- `user_code`:
  - Type: string
  - Required: YES
  - Pattern: Must be exactly 6 digits (`^\d{6}$`)
  - Example: "123456"
  - Source: Generated by user's authenticator app using the secret from `/members/mfa/change`

- `access_token`:
  - Type: string
  - Required: YES
  - Source: Must be the JWT token from `POST /members/signin` response (the `jwt` field)
  - Validation: Should match the Bearer token in the Authorization header

### Response Specification

#### Success Response (201 Created)
```json
{
  "success": true
}
```

**Field Details:**
- `success`:
  - Type: boolean
  - Value: true (indicates MFA is now permanently enabled for the account)

#### Error Responses

**400 Bad Request**
- Triggers:
  - Missing `user_code` field
  - Missing `access_token` field
  - Missing `X-Member-Username` header
  - `user_code` is not exactly 6 digits
- Response: Standard error format with error messages array

**401 Unauthorized**
- Triggers:
  - Invalid or expired JWT token
  - Incorrect `user_code` (TOTP verification failed)
- Response: Standard unauthorized error format

### Backend Verification Checklist
- [ ] Endpoint exists at route `POST /members/mfa/change-verify`
- [ ] Requires authentication via Bearer token
- [ ] Validates `X-Member-Username` header is present
- [ ] Validates `user_code` field is present and matches pattern `^\d{6}$`
- [ ] Validates `access_token` field is present
- [ ] Validates JWT token is valid and not expired
- [ ] Verifies TOTP code against the member's stored secret
- [ ] Permanently enables MFA for the member's account upon successful verification
- [ ] Returns 201 with `success: true` on success
- [ ] Returns 400 if required fields are missing or invalid format
- [ ] Returns 401 if token is invalid/expired or TOTP code is incorrect

## Endpoint 3: POST /members/mfa/validate

### Purpose
Complete the login process for members with MFA enabled by validating their TOTP code.

### Authentication Required
- **Type:** None (public endpoint)
- **Security:** No authentication header required

### Request Specification

#### Headers
```
Content-Type: application/json
```

#### Request Body (JSON)
```json
{
  "username": "string (required)",
  "mfa_code": "string (required)",
  "session": "string (required)"
}
```

**Field Validation:**
- `username`:
  - Type: string
  - Format: email
  - Required: YES
  - Example: "john.smith@goteal.co"
  - Source: Same value used in the original `POST /members/signin` request

- `mfa_code`:
  - Type: string
  - Required: YES
  - Pattern: Must be exactly 6 digits (`^\d{6}$`)
  - Example: "123456"
  - Source: Generated by user's authenticator app

- `session`:
  - Type: string
  - Required: YES
  - Example: "mfa_session_abc123xyz789"
  - Source: The `session` field from the 202 response of `POST /members/signin` (when MFA is enabled)

### Response Specification

#### Success Response (200 OK)
```json
{
  "jwt": "string",
  "member": {
    "id": "string",
    "email": "string",
    "role": "string",
    "name": "string",
    "status": "string",
    "mfa_status": "string"
  }
}
```

**Field Details:**
- `jwt`:
  - Type: string
  - Description: JSON Web Token for authentication
  - Usage: Use in `Authorization: Bearer <jwt>` header for authenticated requests
  - Example: "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."

- `member`:
  - Type: object
  - Required fields: `id`, `email`, `mfa_status`
  - Optional fields: `role`, `name`, `status`
  - Description: Member profile object (same structure as standard signin response)

#### Error Responses

**400 Bad Request**
- Triggers:
  - Missing `username` field
  - Missing `mfa_code` field
  - Missing `session` field
  - `mfa_code` is not exactly 6 digits
- Response: Standard error format with error messages array

**401 Unauthorized**
- Triggers:
  - Invalid or expired `session` token
  - Incorrect `mfa_code` (TOTP verification failed)
  - Username doesn't match the session
- Response: Standard unauthorized error format

### Backend Verification Checklist
- [ ] Endpoint exists at route `POST /members/mfa/validate`
- [ ] Does NOT require authentication (public endpoint)
- [ ] Validates all three fields are present: `username`, `mfa_code`, `session`
- [ ] Validates `mfa_code` matches pattern `^\d{6}$`
- [ ] Validates `session` token is valid and not expired
- [ ] Validates `username` matches the session's associated user
- [ ] Verifies TOTP code against the member's stored secret
- [ ] Generates and returns a JWT token upon successful validation
- [ ] Returns member profile object matching the standard signin response
- [ ] Returns 200 with `jwt` and `member` on success
- [ ] Returns 400 if required fields are missing or invalid format
- [ ] Returns 401 if session is invalid/expired or TOTP code is incorrect

## Endpoint 4 (Updated): POST /members/signin

### Purpose
Authenticate a member and return either a JWT token (if MFA disabled) or a session token (if MFA enabled).

### Authentication Required
- **Type:** None (public endpoint)

### Request Specification

#### Request Body (JSON)
```json
{
  "username": "string (required)",
  "password": "string (required)"
}
```

**Field Validation:**
- `username`: string, required, typically an email
- `password`: string, required, format: password

### Response Specification

#### Success Response - MFA Disabled (200 OK)
```json
{
  "jwt": "string",
  "member": {
    "id": "string",
    "email": "string",
    "role": "string",
    "name": "string",
    "status": "string",
    "mfa_status": "string"
  }
}
```

**When to return 200:**
- User credentials are valid
- User does NOT have MFA enabled
- Returns JWT token for immediate use

#### Success Response - MFA Enabled (202 Accepted)
```json
{
  "session": "string"
}
```

**When to return 202:**
- User credentials are valid
- User HAS MFA enabled
- Returns temporary session token instead of JWT
- Client must call `POST /members/mfa/validate` to complete login

#### Error Responses

**400 Bad Request**
- Triggers:
  - Missing `username` field
  - Missing `password` field
- Response: Standard error format

### Backend Verification Checklist
- [ ] Endpoint exists at route `POST /members/signin`
- [ ] Does NOT require authentication (public endpoint)
- [ ] Validates `username` and `password` are present
- [ ] Checks if user has MFA enabled in their account
- [ ] **IF MFA disabled:** Returns 200 with `jwt` and `member` object
- [ ] **IF MFA enabled:** Returns 202 with `session` token (NOT jwt)
- [ ] Session token is temporary and can be used in `/members/mfa/validate`
- [ ] Returns 400 if required fields are missing
- [ ] Returns appropriate error for invalid credentials

## Data Flow Verification

### Flow 1: MFA Setup (First-Time Configuration)

```
1. POST /members/signin
   Input: { username, password }
   Output (200): { jwt, member }
   ↓
2. POST /members/mfa/change
   Input: 
     - Header: Authorization: Bearer <jwt from step 1>
     - Body: { access_token: <jwt from step 1> }
   Output (200): { secret }
   ↓
3. User adds secret to authenticator app
   ↓
4. POST /members/mfa/change-verify
   Input:
     - Header: Authorization: Bearer <jwt from step 1>
     - Header: X-Member-Username: <email from step 1>
     - Body: { user_code: <6-digit from app>, access_token: <jwt from step 1> }
   Output (201): { success: true }
   
Result: MFA is now enabled for the user
```

### Flow 2: Login with MFA Enabled

```
1. POST /members/signin
   Input: { username, password }
   Output (202): { session }
   ↓
2. POST /members/mfa/validate
   Input: { 
     username: <same as step 1>, 
     mfa_code: <6-digit from app>, 
     session: <session from step 1> 
   }
   Output (200): { jwt, member }
   
Result: User is logged in with JWT token
```

### Backend Verification Checklist - Data Flow
- [ ] JWT token from signin (200) can be used in Authorization header
- [ ] JWT token from signin (200) can be used as `access_token` in MFA setup/verify
- [ ] Session token from signin (202) can be used in MFA validate
- [ ] Session token from signin (202) is temporary and expires appropriately
- [ ] MFA validate returns the same response structure as signin (200)
- [ ] After MFA verify succeeds, subsequent signin returns 202 (not 200)

## Security Requirements Verification

### Token Security
- [ ] JWT tokens have appropriate expiration times
- [ ] Session tokens (for MFA) have shorter expiration than JWT tokens
- [ ] TOTP secrets are stored securely (encrypted/hashed)
- [ ] TOTP verification implements rate limiting to prevent brute force
- [ ] Session tokens are invalidated after successful MFA validation
- [ ] Session tokens are invalidated after expiration time

### TOTP Implementation
- [ ] TOTP implementation follows RFC 6238 standard
- [ ] Time window for TOTP validation is appropriate (typically 30 seconds)
- [ ] TOTP secrets are Base32-encoded
- [ ] TOTP codes are exactly 6 digits
- [ ] TOTP verification allows for time drift (typically ±1 time step)

### Input Validation
- [ ] All email fields are validated for proper email format
- [ ] All required fields return 400 if missing
- [ ] Pattern validation for 6-digit codes is enforced
- [ ] Authorization headers are validated before processing requests

### Error Handling
- [ ] Error responses don't leak sensitive information
- [ ] 401 errors don't distinguish between "invalid token" and "incorrect code"
- [ ] Rate limiting is implemented on MFA verification endpoints
- [ ] Appropriate logging for security events (MFA setup, verify, validate)

## Testing Scenarios

### Scenario 1: Complete MFA Setup Flow
1. Sign in user without MFA → expect 200 with jwt
2. Call mfa/change with jwt → expect 200 with secret
3. Generate TOTP code from secret
4. Call mfa/change-verify with code → expect 201 with success
5. Sign in same user again → expect 202 with session (not 200)

### Scenario 2: Complete MFA Login Flow
1. Sign in user with MFA enabled → expect 202 with session
2. Generate TOTP code from user's secret
3. Call mfa/validate with session and code → expect 200 with jwt
4. Use jwt for authenticated requests → expect success

### Scenario 3: Error Cases
1. Call mfa/change without Authorization header → expect 401
2. Call mfa/change with invalid jwt → expect 401
3. Call mfa/change-verify with wrong TOTP code → expect 401
4. Call mfa/change-verify without X-Member-Username header → expect 400
5. Call mfa/validate with expired session → expect 401
6. Call mfa/validate with wrong TOTP code → expect 401
7. Call mfa/validate with mismatched username → expect 401

### Scenario 4: Field Validation
1. Call mfa/change without access_token field → expect 400
2. Call mfa/change-verify with 5-digit code → expect 400
3. Call mfa/change-verify with non-numeric code → expect 400
4. Call mfa/validate without session field → expect 400
5. Call mfa/validate with invalid email format → expect 400
